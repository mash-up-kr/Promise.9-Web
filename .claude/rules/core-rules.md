## 프로젝트 절대 규칙 (항상)
1. **pnpm만** 사용. npm / yarn 금지.
2. 명령 실행 전 **현재 디렉터리 확인**. 루트와 `extension/` 은 `package.json` 이 다르다.
3. **컴포넌트·훅·유틸·타입을 새로 만들기 전 반드시 기존 코드를 먼저 검색**한다(중복 금지) → `/check-dup`.
4. import 는 절대경로 `@/`(앱/웹) · `@shared/`(공유 코어). `../../` 상대경로는 같은 기능 폴더 내부에서만.
5. 스타일은 NativeWind `className`. inline StyleSheet 는 동적 값 등 불가피할 때만.
6. 커밋 메시지는 Conventional Commits (`feat:`, `fix:`, `chore:` ...). 본문 한국어 OK.
7. **테스트 먼저 (TDD).** 새 동작·기능 코드는 **실패하는 테스트를 먼저** 쓴다(red) → 통과시키는 **최소** 구현(green) → 정리(refactor). 구현 코드를 먼저 짜지 않는다. 상세·패턴: docs/conventions/testing.md. (E2E 는 별도 담당 — unit·integration 만.)

# 행동 원칙

LLM 코딩 실수를 줄이기 위한 행동 가이드라인.

**Tradeoff:** 이 가이드는 속도보다 신중함에 무게를 둔다. 사소한 작업엔 판단껏 한다.

## 1. Think Before Coding
**추측하지 말 것. 혼란을 숨기지 말 것. 트레이드오프를 드러낼 것.**

구현 전에:
- 가정을 명시한다. 불확실하면 묻는다.
- 해석이 여러 개면 모두 제시한다 — 말없이 하나만 고르지 않는다.
- 더 간단한 방법이 있으면 말한다. 필요하면 반대 의견을 낸다.
- 불명확하면 멈춘다. 무엇이 혼란스러운지 짚고 질문한다.
- 읽지 않은 코드를 추측하지 않는다. 주장하기 전에 파일을 읽고 ripgrep 으로 사용처를 확인한다.

## 2. Simplicity First
**문제를 푸는 최소 코드. 추측성 코드 없음.**

- 요청 범위 밖의 기능 추가 금지.
- 1회용 코드에 추상화 금지.
- 요청하지 않은 "유연성 / 설정 가능성" 금지.
- 일어날 수 없는 시나리오를 위한 에러 처리 금지.
- 200줄 짰는데 50줄로 가능하면 다시 쓴다.

스스로 묻는다: "시니어 엔지니어가 이걸 과하다고 할까?" → 그렇다면 단순화한다.

## 3. Surgical Changes
**꼭 필요한 것만 건드린다. 내가 만든 흔적만 치운다.**

기존 코드를 수정할 때:
- 인접 코드·주석·포맷을 "개선"하지 않는다.
- 망가지지 않은 것을 리팩터링하지 않는다.
- 내 취향과 달라도 기존 스타일을 따른다.
- 관련 없는 죽은 코드를 발견하면 — 삭제하지 말고 언급한다.

내 변경이 고아(orphan)를 만들면:
- 내 변경 때문에 안 쓰이게 된 import / 변수 / 함수는 제거한다.
- 기존부터 있던 죽은 코드는 요청 없이 제거하지 않는다.

기준: 바뀐 모든 줄이 사용자의 요청과 직접 연결돼야 한다.

## 4. Goal-Driven Execution
**성공 기준을 정의하고, 검증될 때까지 반복한다.**

작업을 검증 가능한 목표로 바꾼다:
- "검증 추가" → "잘못된 입력에 대한 테스트를 쓰고, 통과시킨다"
- "버그 수정" → "버그를 재현하는 테스트를 쓰고, 통과시킨다"
- "X 리팩터" → "전후로 테스트가 통과하는지 확인한다"

다단계 작업은 간단한 계획을 먼저 밝힌다:

```
1. [단계] → 검증: [확인]
2. [단계] → 검증: [확인]
3. [단계] → 검증: [확인]
```

명확한 성공 기준이 있으면 독립적으로 반복할 수 있다. 약한 기준("그냥 되게 해줘")은 계속 되묻게 만든다.

# 기능 개발 워크플로

사용자가 "이런 기능 개발해줘" 류로 **기능 구현을 명령**하면 아래 흐름을 따른다.

1. (필요 시) `/check-dup` 으로 중복 확인.
2. 기능을 **관심사별 atomic 커밋 단위**로 쪼개서 개발한다.
3. 각 단위는 **TDD 로** 만든다: **실패 테스트 먼저(red) → 최소 구현(green) → 정리(refactor)**. 테스트 작성은 `/new-test`. (절대 규칙 7)
4. 각 커밋 단위마다 **테스트 통과 + lint·타입 검증 후 알아서 커밋**한다(Conventional Commits). 매 커밋마다 사용자에게 따로 묻지 않는다.
5. 작업 완료 후 **PR 생성 직전에만** 사용자에게 확인받는다 → 승인 시 `/pr`.

지키는 경계:
- 현재 브랜치가 `main` 이면 **커밋 전에 작업 브랜치부터 생성**한다.
- 자동 진행은 커밋까지다. **PR 생성**, 그리고 force push·브랜치 삭제·hard reset 등 파괴적 작업은 별도 확인을 유지한다(아래 "검증 & 안전").
- 작업 도중 **설계상 갈림길**(컴포넌트 위치, 라이브러리 선택, 모호한 요구사항)은 커밋 게이트와 무관하게 멈추고 묻는다("Think Before Coding").

## 검증 & 안전 (보강)
- "완료" 선언 전: 요구사항 재확인, 테스트·lint 실행, 무엇을 바꿨고 무엇을 검증 못 했는지 명시한다.
- 테스트가 실패하면 출력과 함께 그대로 보고한다. 숨기지 않는다.
- 파괴적 작업 전 확인한다: 파일/브랜치 삭제, force push, hard reset, `--no-verify`.
- 의존성 없는 도구 호출은 병렬로, 의존적인 것은 순차로 실행한다.

---

**이 가이드가 작동 중인 신호:** diff에 불필요한 변경이 줄고, 과설계로 인한 재작성이 줄고, 실수 후가 아니라 구현 전에 질문이 나온다.
